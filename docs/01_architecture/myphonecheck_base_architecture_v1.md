# MyPhoneCheck Base Architecture v1

## 2. Purpose

이 문서는 MyPhoneCheck 제품의 **기준(base) 아키텍처**를 선언한다. 대표님 구두 합의와 구조화된 정리본을 단일 출처로 묶으며, 이후 디벨롭·문서·검증의 **상위 기준**으로 사용한다.

이 문서는 구현 상세나 코드 레벨 설계를 규정하지 않는다. 제품·시스템 아키텍처 레벨에서의 합의를 고정한다.

---

## 3. Product Definition

MyPhoneCheck는 **모르는 전화번호·문자 발신 번호**에 대해 사용자가 **즉시 판단**할 수 있도록 돕는 **결정 지원(decision-support)** 앱이다.

앱은 **네 가지 체크 유닛**으로 구성된다.

- Call Check  
- Message Check  
- Mic Check  
- Camera Check  

Google Play Console 지원 **190개국 이상 동시 론칭**을 목표로 한다.

---

## 4. Core Philosophy

- 앱은 **증거를 모아** 사용자가 **받을지 말지** 결정하게 돕는다.
- 앱은 **최종 결정을 대신하지 않는다.**
- **외부 검색 결과**와 **내부 디바이스 이력**을 요약해 **판단 근거**를 제공한다.
- **국가별 차이**는 **정책 레이어**와 **표현(Presentation) 레이어**에서 처리하고, **판단 코어**는 **최대한 공통 구조**로 유지한다.

---

## 5. Information Architecture

상위 화면 구조는 다음을 둔다.

- **Home** — 앱의 진입·요약·유닛 진입 허브  
- **Settings** — 언어·국가·권한·구독·개인정보 등 전역 설정  
- **각 체크 유닛 상세** — 해당 유닛의 **히스토리·타임라인**을 포함한 전용 화면  

---

## 6. Unit Architecture

### 6.1 Call Check

- 전화 **수신 시** **2~3초 내** **오버레이**로 결과를 표시한다.
- **외부 검색 결과**, **내부 통화·문자 이력**, **앱 내부 태그**를 함께 표시한다.
- 사용자는 **수신·거절·차단**을 선택할 수 있다.

### 6.2 Message Check

- 문자 **수신 후** **2~3초 내** 결과를 표시한다.
- 판단은 **발신 번호**를 중심으로 한다.
- 사용자는 **수신·거절·차단·태그 남기기**를 수행할 수 있다.

### 6.3 Mic Check

- **마이크 권한을 보유한 앱 목록**을 제공한다.
- **최근 사용 기록**을 제공한다.
- **권한 보유 여부**, **차단 기능**, **사용 기록**을 다룬다.
- **앱 사용 기록**과 **사용자 직접 사용 기록**을 구분해 제공한다.

### 6.4 Camera Check

- **카메라 권한을 보유한 앱 목록**을 제공한다.
- **최근 사용 기록**을 제공한다.
- **권한 보유 여부**, **차단 기능**, **사용 기록**을 다룬다.
- **앱 사용 기록**과 **사용자 직접 사용 기록**을 구분해 제공한다.

---

## 7. Tag System

- 사용자는 번호를 **연락처에 저장하지 않고도** 앱 내부에서만 **관계·맥락**을 남길 수 있다.
- 이 레이어는 **외부 주소록·메신저 동기화**와 **분리**된 **앱 내부 전용** 구조다.
- 이후 동일 번호 **수신 시** **외부 검색**, **내부 통화·문자 이력**, **과거 태그 기록**을 **한 화면의 근거 묶음**으로 제시한다.

---

## 8. Initial Scan Model

- 앱 **설치 직후** 권한이 부여되면 **즉시 초기 스캔**을 수행한다.
- 스캔 대상은 다음을 포함한다.
  - 전화번호·통화 이력  
  - 문자 이력  
  - 마이크 권한 및 사용 기록  
  - 카메라 권한 및 사용 기록  
- 초기 스캔 결과는 이후 판단·표시의 **앱 내부 베이스 데이터**가 된다.

---

## 9. Layered Architecture

시스템은 아래 네 층으로 기술한다.

| Layer | 역할 |
|--------|------|
| **Presentation Layer** | 화면·복사·톤·로캘 표현. 판단 코어를 재정의하지 않는다. |
| **Decision / Domain Layer** | 번호 단위 판단·카테고리·추천 행동·태그·정책 해석의 중심. |
| **Data Layer** | 온디바이스 저장·캐시·검색 증거 수집 결과 등의 보존·조회. |
| **Device Integration Layer** | 통화 스크리닝·문자·권한·시스템 API·디바이스 이벤트와의 연결. |

`docs/01_architecture/global-single-core-system.md`에 기술된 **Global Core·Country Policy·Presentation** 3분할과 충돌하지 않는다. 본 문서의 4층은 **구현·리포지토리 정렬**을 위한 기준 축이며, **국가 정책·표현 분리** 원칙은 Decision/Data와 Presentation 사이의 경계에서 유지한다.

---

## 10. Global Expansion Model

- **190개국 이상** 동시 론칭을 전제로 한다.
- **디바이스 중심 적응형** 구조를 취한다.
- **언어·로케일·전화번호 표기·검색 플랫폼 우선순위**는 **디바이스와 국가 맥락**을 기준으로 처리한다.
- **국가별 1등·2등 검색 플랫폼** 결과를 활용한다.

---

## 11. Subscription Model

- **전 세계 단일 요금제** 방향을 취한다.
- 가격대는 **약 1.5달러 수준**을 선호한다.
- **첫 1개월 무료**를 제공한다.
- **31일째부터** 과금을 시작한다.
- **동일 디바이스**의 무료 기간은 **1회 1개월**로 고정한다.
- **가입**과 **탈퇴**는 **동일한 난이도·동일한 노출성**을 유지한다.

---

## 12. Architectural Principles

- **디바이스 중심:** 판단 근거는 측정 가능한 온디바이스 데이터를 우선한다.
- **글로벌 단일 코어:** 판단 파이프라인은 국가별로 갈라지지 않는 공통 축을 유지한다.
- **국가 차이는 정책·표현:** 규제·검색 소스·문구는 정책·표현 레이어에 둔다.
- **외부 검색 + 내부 이력 + 앱 내부 태그 결합:** 단일 번호 프로필에 근거를 합성한다.
- **제로 비용·제로 저장 원칙:** 기존 제품 문서와의 정합을 **재검토**한다. (§13)
- **네오(로컬 OS 에이전트) 중심 운영·검증:** 빌드·디바이스·리포트 파이프라인을 **정렬**한다. (§13)

---

## 13. Open Questions / To Be Developed Next

다음 항목은 본 기준 문서에 **선언만** 하고, 상세 규칙·구현은 후속 디벨롭에서 확정한다.

1. **제로 비용·제로 저장 원칙**과 본 문서의 데이터·검색·구독 모델 간의 **정의 정합**을 재검토한다.  
2. **네오 중심** 운영·검증(로컬 에이전트, 리포지토리 게이트, 캡처·로그)과 앱 릴리스 프로세스의 **접점**을 문서·도구 양쪽에서 정렬한다.  
3. Call/Message와 Mic/Camera 유닛 간 **공통 Decision 레이어**와 **권한·프라이버시 UX**의 경계를 제품 스펙으로 구체화한다.
